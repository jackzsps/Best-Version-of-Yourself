# Troubleshooting iOS Actions & TestFlight Uploads

When working with iOS projects, modifying the `PRODUCT_BUNDLE_IDENTIFIER` or attempting to upload a build to TestFlight can result in several complex errors. Use this workflow to diagnose and resolve them.

## 1. App Name Already in Use (Bundle ID Conflicts)

**Error Message:**
`App Record Creation failed due to request containing an attribute already in use. The app name you entered is already being used for another app in your account.`

**Why this happens:**
App Store Connect links an app's "Name" to its "Bundle ID". If the `PRODUCT_BUNDLE_IDENTIFIER` in `project.pbxproj` is modified but the project attempts to use an existing App Name on App Store Connect, Apple treats it as a brand new app requesting an already-claimed name.

**How to resolve:**
1. **To update an existing TestFlight app:** You **must** revert the Bundle ID in both the iOS (`project.pbxproj`) and Android (`app/build.gradle`) projects back to the original Bundle ID that was first uploaded to App Store Connect.
2. **To keep the new Bundle ID:** Go to App Store Connect, find the old app, rename it (e.g., append "Old"), save, and then run the upload command again to register the new Bundle ID with the desired App Name.

## 2. Apple Sign-In Failing (`auth/invalid-credential`)

**Error Message:**
Firebase returns `auth/invalid-credential` on `.signInWithCredential()` for Apple Login, while Google/Email login works perfectly.

**Why this happens:**
Firebase relies on the Apple Sign-In identifier token. If the Bundle ID of the app was recently changed, the tokens generated by the device no longer match the expected configuration in the Firebase Project and Apple Developer Console.

**How to resolve:**
1. **Verify Bundle IDs match perfectly:** Ensure that the `PRODUCT_BUNDLE_IDENTIFIER` in Xcode exactly matches the iOS Bundle ID registered in the Firebase Console.
2. **Update `GoogleService-Info.plist`:** Download a fresh `GoogleService-Info.plist` from Firebase containing the correct Bundle ID and replace the existing one in the project.
3. **Check Apple Developer Settings:** Ensure the App ID in the Apple Developer portal has the "Sign In with Apple" capability securely checked.

## 3. Empty CFBundleVersion Error

**Error Message:**
`CFBundleVersion is empty but must be composed of one to three period-separated integers.`

**Why this happens:**
Xcode expects a valid version number for `CFBundleVersion`, often parameterized via `$(CURRENT_PROJECT_VERSION)`. If tools like Fastlane (`increment_build_number`) or Expo EAS fail to correctly inject or mutate the Build Number, during the archive/export phase, the value evaluates to an empty string.

**How to resolve:**
1. Check if `CURRENT_PROJECT_VERSION` is defined and populated properly inside `project.pbxproj`.
2. As an immediate fix to unblock CI or Xcode uploads, hardcode `CFBundleVersion` inside `Info.plist` to a fallback integer (e.g., `14` if previously at `13`). Avoid parameterizing it if build toolchains fail to substitute the variable.

## 4. App Encryption Documentation Prompt

**Error/Prompt Message:**
`App Encryption Documentation: What type of encryption algorithms does your app implement?` (In App Store Connect / TestFlight)

**Why this happens:**
Apple requires all apps to declare whether they use standard or custom encryption for US Export Compliance purposes. By default, every new build uploaded to TestFlight will be paused asking this question, even if you only make standard HTTPS calls.

**How to resolve:**
To automatically bypass this prompt for apps that don't need special compliance (like using standard HTTPS):
1. Open `ios/YourAppName/Info.plist`.
2. Add the `ITSAppUsesNonExemptEncryption` key and set it to `<false/>`.
```xml
<key>ITSAppUsesNonExemptEncryption</key>
<false/>
```
